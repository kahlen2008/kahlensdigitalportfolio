Project Journey

01

The Problem

Client's challenge and project scope

Singapore's ageing population faces increasing social isolation. Community centres needed a structured digital system to manage intergenerational engagement programmes, but relied on manual, error-prone coordination processes.

Core Challenge

No structured system for event coordination—manual tracking led to double-bookings, missed registrations, and lost community suggestions.

Project Scope

Build a complete Event & Booking Management System with admin moderation, user registration, and a community-driven suggestion pipeline.

02

My Role

Responsibilities and involvement

As one of the full-stack developers on Bridgify, I was assigned ownership of the Event & Booking Management System — the platform's most critical module. I handled the full feature lifecycle for this module, from database schema design to frontend interactivity.

Backend Architecture Frontend Development Database Design System Design Admin Logic Form Validation
Backend (Python & Flask)

Designed Flask routes for event CRUD, booking management, suggestion submissions, and admin approval workflows. Each route follows clean request handling: receive form data → validate input → execute SQL → commit changes → redirect.

Frontend (HTML, CSS, JS, Jinja)

Built modal-based UI for event creation, suggestions, and admin review. Used Jinja templating for dynamic data rendering and JavaScript for smooth interactions without full page reloads.

03

Solution Design

Thinking process and architecture decisions

I designed an end-to-end workflow that transforms community suggestions into published events through a structured moderation pipeline.

User Submits Suggestion

Community members propose event ideas through a modal form. Submission is stored with "Pending" status.

Admin Reviews in Dashboard

Administrators access a dedicated suggestions panel to review, approve, or reject proposals.

Auto-Fill Event Creation

Upon approval, the title and description auto-populate the event creation form via hidden input fields and Jinja template variables.

Event Published to Calendar

Admin sets date, time, and location. Event commits to the database and appears on the user calendar immediately.

Database Architecture

Implemented a relational SQLite database with separate tables for Events, Bookings, and Suggestions. Each table was structured with appropriate fields and relationships to maintain data consistency across the pipeline.

Validation & Integrity

Built server-side form validation to prevent partial or invalid submissions. Implemented date/time constraints so past events cannot be created, and end times must follow start times.

04

Work Process

Execution, challenges, and problem-solving

Building a production-style system required iterative development, debugging, and adapting to unexpected constraints.

Development Approach

I followed an iterative process: build core CRUD first, then layer on the suggestion workflow, admin panel, and validation logic. Each feature was tested independently before integration.

Challenges Encountered

Database migration issues when restructuring tables, modal state management across multiple forms, and ensuring real-time updates reflected correctly across all view states without full page reloads.

Problem-Solving

Implemented a custom DatabaseHelper class to abstract SQL operations, designed a robust status-tracking system for suggestions, and built cascading UI updates to ensure frontend-backend synchronisation.

Quality Assurance

Rigorous form validation including date constraints, slot availability checks, and input sanitisation. Admin actions are access-controlled with password authentication.
